# Enriched-Data Deep Dive — What the Professor's Extra Data Tells Us About `toxic_data_01111`

**Goal:** maximize F1 (positive class) on the held-out `toxic_exam_01111` set.
**TL;DR:** The enriched dataset let us fully **reverse-engineer the data generator**. This is extremely useful for *understanding* the task and for *confidence in feature selection*, but it does **not** raise the achievable score — `01111`'s F1 is already near its ceiling (~**0.65**) and that ceiling is set by signals that live **inside** `toxic_data_01111` itself, not in the extra files.

---

## 1. What the enriched download actually contains

`data/download/` holds four families, each spanning **all 32 five-bit IDs** (`00000`…`11111`):

| Family | Rows (train/exam) | Columns | Label col | Missing values | Injected corruptions |
|---|---|---|---|---|---|
| **Toxic_Data / Toxic_Exam** *(your task)* | 8 000 / 2 000 | `f00`…`f15` | `label` | Yes (f11, f15) | MNAR missingness + linear scaling + heavy-tail |
| **Lethal_Data / Lethal_Exam** | 24 000 / 6 000 | `feat_00`…`feat_15` | `target` | **None** | **None** (clean `N(0,1)`) |

So **Toxic** and **Lethal** are two *parallel* tracks generated by the same underlying label rule but with **different difficulty**. Your graded assignment is **Toxic `01111`**.

---

## 2. The generator, decoded

### 2.1 Rows are independent across IDs
The same row index in two different datasets is **unrelated** (f12 correlation `00000` vs `01111` ≈ 0.014; label agreement = chance). → **No cross-dataset row lookup is possible.** The value of the extra data is *structural*, not *row-level*.

### 2.2 The ID's first two bits choose the label-driving feature
Across all 32 datasets, the single feature carrying the label signal is fixed by the **first two ID bits**:

| First 2 bits | Driver feature |
|---|---|
| `00` | f04 |
| `01` | **f12** |
| `10` | f08 |
| `11` | f00 |

`01111` starts with `01` → **the causal driver is `f12`.** This was confirmed independently in *both* the Toxic and the Lethal `01111` files (Lethal's `feat_12` is also its driver), so the rule is a genuine property of the generator, not a coincidence.

### 2.3 Shape of the label function `g(f12)` differs by track
- **Toxic `01111`:** positive rate is a **skewed U-shape** in `f12` — ~3.8% near `f12≈0`, rising to **44.9%** at `f12≈+2.06` and 13.7% at `f12≈−2.06` (steeper on the positive side).
- **Lethal `01111`:** positive rate is **monotone increasing** in `feat_12` (6.6% → 20.4%), no missingness leak to help.

The label is **probabilistic** — even the most-positive `f12` bin is only ~45% positive. This is the fundamental reason single-feature prediction is weak and why there is a hard F1 ceiling.

### 2.4 The Toxic corruption signature (per dataset)
The first two ID bits also select an 8-feature window that gets corrupted with a rotating pattern `[Missing · Heavy-tail · Heavy-tail · Missing · clean · Scaled · Scaled · Scaled]`. For **`01111`** this lands as:
- **MNAR missingness:** `f11`, `f15`
- **Heavy-tail:** `f09`, `f15`
- **Linear scaling:** `f10`, `f13`, `f14`
- **Untouched clean `N(0,1)`:** `f00`–`f08` and **`f12`** (your driver stays clean — good).

## 3. Where `01111`'s predictive signal really comes from (ablation, 5-fold OOF, XGBoost)

| Feature set | CV F1 |
|---|---|
| Missingness flags only (`f11_missing`, `f15_missing`, both) | 0.36 |
| + `f12` (the causal driver) | 0.42 |
| + scaled `f10/f13/f14` | 0.46 |
| + `f11` value, `f15`/`f09` signed-log | 0.56 |
| **Full engineered pipeline (current `train_model.py`)** | **0.654** |

Two of these signals are **label leaks the professor injected**, and they are the strongest levers:

1. **MNAR missingness leak (biggest single signal).** Positives are far more likely to have a missing value:
   - `f11` missing → **26.8%** positive vs **6.9%** when present
   - `f15` missing → **26.4%** positive vs **7.0%** when present
2. **`f11` present-value leak.** Among rows where `f11` is present, the value itself correlates **−0.115** with the label — a second, independent leak.
3. **`f12` (causal driver)** adds a modest amount on top (MI ≈ 0.035; it is weak because the label is noisy).

> All of these live **inside** `toxic_data_01111`. The extra datasets cannot strengthen them.

---

## 4. Does the enriched / Lethal data improve the score? (the decisive test)

We learned `g(f12)` from **256 000 pooled aligned rows** across all other datasets and fed it into the real `01111` pipeline:

| Configuration | CV F1 |
|---|---|
| Internal engineered pipeline (baseline) | **0.6538** |
| Baseline **+ external `g(f12)`** feature | 0.6407 ⬇ |
| External `g(f12)` **replacing** internal `f12` | 0.6126 ⬇ |

**Conclusion: external data does not help — it slightly hurts.** Reasons:
- The `f12`→label relationship is already learned perfectly well from the 8 000 internal rows (the driver signal saturates).
- `01111`'s score is **dominated by the internal MNAR missingness leak**, which no external dataset can augment.
- The external parametric `g` is a hair less flexible than what XGBoost extracts internally.

Lethal `01111` is also no help: different schema, **no missingness leak**, only a weak monotone `feat_12` signal → its own F1 ceiling is far lower (~0.25–0.30). It is a harder, separate track, not extra training data for your task.

---

## 5. What this means for maximizing your score

### Keep doing (these are the real levers)
1. **Missingness indicators are king.** Always include `f11_missing`, `f15_missing`, and `both_missing`. These alone give F1 ≈ 0.36 and anchor the model.
2. **`f11` present-value** carries an independent leak — keep `f11` (median-imputed) as a feature.
3. **`f12` nonlinear transforms.** Because `g(f12)` is a skewed-U, add `f12**2`, `abs(f12)`, and `f12*sign`. Tree models approximate this, but explicit terms help linear/MLP members of an ensemble.
4. **Signed-log heavy-tail features** (`f15`, `f09`) and the **scaled** `f10/f13/f14` add small but real lift — keep them.
5. **Tune the decision threshold on OOF predictions** (current pipeline sweeps it). With ~10% positives and F1 as the metric, the optimal threshold is well below 0.5 (≈0.6 of the XGB probability after `scale_pos_weight=9`). This is worth several F1 points.
6. **`scale_pos_weight ≈ 9`** (= neg/pos ratio) in XGBoost is important for F1.

### Don't bother with (proven dead ends)
- ❌ Adding the enriched/external datasets as extra training rows or as a learned `g(f12)` feature — no lift, slight loss.
- ❌ Using Lethal data for the Toxic task — different generator, no shared leaks.
- ❌ Cross-dataset row lookups — rows are independent.

### Realistic ceiling
The label is **intrinsically probabilistic** (best `f12` bin only ~45% positive). Combining the missingness leak + `f11`-value leak + `f12` driver, the honest, reproducible CV ceiling is **F1 ≈ 0.65**. Pushing meaningfully past this is unlikely; effort is better spent on **threshold calibration and a small, stable ensemble** than on new data.

---

## 6. One-line answer to "what's the difference between Toxic and Lethal?"
**Toxic** = your task: smaller, deliberately *corrupted* data whose injected missingness is a strong, exploitable label leak (ceiling ≈ 0.65). **Lethal** = a larger, *clean* parallel track (`feat_*`/`target`, no missingness, no injections) where the same driver feature acts monotonically but weakly — a genuinely harder problem and **not usable** for the Toxic `01111` submission.
